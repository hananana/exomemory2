# Plan: v0.9 — wiki quality (confidence + supersession + typed links + MERGE conflict)

agentmemory (rohitg00) が Karpathy LLM Wiki の弱点として挙げた 4 点 — confidence 管理 / 古い情報の腐敗 / wikilink の意味欠如 / ingest の揺れ — を、exomemory2 の "純 Markdown + Dataview" 哲学を壊さずに取り込む。

スキーマと ingest workflow を一括で v0.4 → v0.9 へ bump する。

> **#4 quality lint と #5 decay は本リリースに含めない**。理由は末尾 "Future work" 参照。

## Goals

1. **#1 confidence scoring**: entity / concept ページの frontmatter に `confidence` / `sources` / `last_verified` を導入。CREATE / MERGE で自動更新。低 confidence ページの dashboard を提供
2. **#2 supersession**: 「X から Y へ移行した」事実を frontmatter で表現できるようにする (`supersedes` / `stale` / `superseded_by` / `superseded_at`)。`/wiki-query` は default で stale を除外
3. **#3 typed relationships**: Connections セクションを Dataview inline field 化 (`depends_on::` / `contradicts::` / `caused_by::` / `fixed_in::` / `supersedes::` / `related_to::`)。typed link 用 dashboard 追加
4. **#6 MERGE conflict resolution**: WIKI.md の MERGE rule に明示的な決定木を書く (supersession / recency / authority / dual-statement)。ingest の再現性を上げる
5. **`/wiki-migrate` 拡張**: 既存 vault の entity / concept に confidence + sources を backfill、Connection 行を `related_to::` プレフィックス化、新 dashboards を配布

## Non-goals

- **#4 quality lint コマンド** (`/wiki-lint`) — confidence の運用データが十分に溜まってから v0.10 で別出し
- **#5 decay / forgetting** — vault がまだ小さく、設計に必要な経験則が足りない (議論済み、別途 v1.x 検討)
- **既存 wiki ページの Connection 行を typed に "賢く分類" する自動 retrofit** — 機械的な意味推定は危険なので、`related_to::` への一律プレフィックス化のみ。新規 ingest 以降は Claude が typed で書く
- **stale ページの自動 archive / 物理削除** — frontmatter フラグだけ立て、ファイルは残す
- **BM25 / vector hybrid search (#7)、4-tier consolidation (#8)** — exomemory2 の "no DB / 純 Markdown / Karpathy 2 層" 原則を破るので採用しない

## Context

### #1 confidence の根拠

現在の entity / concept ページは「1 個の handover でしか言及されてない弱い知識」と「10 件の独立ソースで裏取りされた強い知識」が同じ顔をして並ぶ。`/wiki-query` 時に重み付けできないし、ユーザーが Obsidian で開いても見分けがつかない。

agentmemory の confidence scoring を Markdown 互換の形で持ち込む。値そのものは導出 (sources count から決定) し、ユーザーが手で書く必要はない。

### #2 supersession の根拠

Karpathy 原典の最大の弱点 (Jônadas Techio, Mehul Gupta が Web で指摘) は「古い情報がそのまま残ること」。exomemory2 でも例えば「以前は X ライブラリを使っていたが Y に乗り換えた」というケースで、X の entity ページが古い記述のまま生き続ける。raw/ は immutable なので raw を消す手は使えず、wiki/ 側で stale 印を立てる必要がある。

### #3 typed links の根拠

現在 `[[link]]` は「繋がっている」しか表現しない (Penfield Labs が言う 1bit 情報)。Dataview の inline field 構文 (`key:: value`) を Connections セクションで使えば、純 Markdown のまま意味が乗る。Dataview なしのエディタでも `- depends_on:: [[X]]` は普通の bullet として読める (degrade gracefully)。

### #6 MERGE conflict resolution の根拠

現状の WIKI.md の MERGE rule は「矛盾時は dated note で両論併記」とだけ書かれており、Claude の判断に揺れが残る。同じ raw を 2 回 ingest すると微妙に違う内容が出る。決定木を WIKI.md に明文化することで再現性を上げる。

## Design decisions

### D1 (#1): confidence は sources count からの純関数

**Frontmatter spec (entity / concept page)**:

```yaml
---
title: <canonical name>
type: entity | concept
tags: [...]
last_updated: YYYY-MM-DD
confidence: 0.6                 # v0.9+, 0.0-1.0, derived from sources + contradictions
sources: 3                      # v0.9+, count of distinct source pages connected
last_verified: YYYY-MM-DD       # v0.9+, last MERGE/CREATE date
stale: false                    # v0.9+, optional, only present when true
supersedes: [[old-page]]        # v0.9+, optional, set on superseder
superseded_by: [[new-page]]     # v0.9+, optional, set on superseded
superseded_at: YYYY-MM-DD       # v0.9+, optional, paired with superseded_by
---
```

**confidence 計算式 (deterministic)**:

```
base = clamp(sources / 5.0, 0.3, 1.0)
# 1 source → 0.3, 2 → 0.4, 3 → 0.6, 4 → 0.8, 5+ → 1.0

if contradictions_present:
  confidence = min(base, 0.5)   # 矛盾検出ページは 0.5 で頭打ち
else:
  confidence = base
```

`contradictions_present` の定義 (**ingest と migrate で完全に同一の判定**を使う):

> 当該 entity / concept ページの `## Connections` セクション内に、行頭が `- contradicts::` で始まる bullet が **1 行以上**存在する場合 true、そうでなければ false。Connections セクションが無い、または `contradicts::` 行が無いなら false。

これにより v0.9 で導入する typed links の `contradicts::` キーが confidence の唯一のシグナルになる。Claude が ingest / MERGE 時に Connections に `- contradicts:: [[X]]` を追加すると、その瞬間に confidence cap が発動する。stdlib Python の行ベース grep で同じ判定ができ、ingest 時 / migrate 時の値が確実に一致する。

**stale=true のページも confidence 計算対象**で、stale 印が付くだけで confidence は減らさない (ユーザーが過去経緯として参照することがあるため)。

> **互換性メモ**: 既存の Source Page Format には別途 `## Contradictions` セクションが存在するが、これは **source ページ専用** (raw → wiki/sources/ にコピーした際の矛盾メモ用)。entity / concept の confidence 計算には影響しない。混同を避けるため、WIKI.md の D4 決定木更新時にこの区別を Notes として明記する。

**`sources` は何をカウントするか — grep ベースで定義を一本化**:

そのページ slug への `[[wikilink]]` を持っている `wiki/sources/*.md` の数。**Dataview には依存しない** (Dataview は実行時計算で shell/python から到達できず、定義揺れの原因になる)。

ingest と migrate **両方で同じ計算式**を使う:

```bash
sources_count() {
  local vault="$1" slug="$2"
  # [[<slug>]] で完全一致するリンクを wiki/sources/ 配下で grep。
  # [[<slug>|alias]] や [[<slug>#anchor]] にもヒットするよう regex を使う。
  grep -l -E "\[\[${slug}(\||#|\])" "${vault}"/wiki/sources/*.md 2>/dev/null | wc -l | tr -d ' '
}
```

これにより:
- ingest 時 (Claude が新 source を書いた直後の MERGE): 同関数で再カウント
- migrate 時 (backfill): 同関数で計算
- 結果が **純関数**で **idempotent**

既存 dashboard "popular-entities" は別系で `length(file.inlinks)` を引き続き使う (Dataview リアルタイム表示用)。frontmatter の `sources` は ingest/migrate 時点の snapshot という位置づけ。両者の値が短期的にズレることは許容する (リアルタイム性が要るのはダッシュボードだけ、confidence 計算は ingest snapshot で十分)。

### D2 (#2): supersession は明示的な linguistic trigger でのみ発動

**自動検出は控えめに**: 「X is better than Y」「X uses Y」のような曖昧な関係まで supersession 化すると暴発する。Claude が ingest の際に **以下のフレーズを source 中に検出した場合のみ** supersession を発動する:

- "switched from X to Y" / "moved from X to Y"
- "X was replaced by Y" / "replaced X with Y"
- "deprecated X in favor of Y"
- "X is no longer used, we use Y now"
- 日本語: 「X から Y に移行」「X をやめて Y にした」「X を Y で置き換え」「X は廃止」

これら以外の「単に新しい方が出てきた」ケースでは supersession を立てず、両ページに `related_to::` で繋ぐだけにする。

**Frontmatter 操作**:

```yaml
# 新ページ (Y, superseder):
supersedes: [[X]]

# 旧ページ (X, superseded):
stale: true
superseded_by: [[Y]]
superseded_at: 2026-04-29
```

旧ページの本文・Connections は **保持する**。stale フラグを立てるだけで、情報は捨てない。

**`/wiki-query` の stale フィルタ — 機械的 retrieval rule で定義する**:

LLM 判断ではなく `wiki/index.md` から候補をピックする pre-retrieval フェーズで、以下のルールを **deterministic に適用**する。

候補ページは以下のいずれかを満たすときのみ context に投入する:

| Rule | 判定 | 説明 |
|------|------|------|
| **R-1** (default include) | `stale != true` | stale フラグの無いページは常に include |
| **R-2** (history keyword) | 質問文 (raw text) に history / 経緯 / 以前 / 昔 / deprecated / 古い / previous / old / 廃止 / なぜ…やめた のいずれかを **大小文字無視で部分一致**で含む | 該当時は **stale も含めて全候補 include** |
| **R-3** (direct page reference) | 質問文に該当ページの slug が `[[<slug>]]` リテラルで含まれる、**または** ページ title 文字列が完全一致で含まれる | 該当ページは stale でも include |
| **R-4** (supersession traversal) | R-1〜R-3 で include された non-stale ページの frontmatter に `supersedes: [[X]]` がある場合、X のページも 1 ホップ追加で include | 「X は何に置き換わった？」「X の後継は？」型クエリで stale 側を拾うため |

判定は Claude **ではなく** wiki-query の Bash プリプロセス (grep + 単純文字列検索) で行う。これにより stale フィルタが LLM 判断に揺れず、retrieval 漏れによる誤答が防げる。Claude が context を読んだ後の応答生成は従来通り。

ルール表は wiki-query.md の Step に明示し、frontmatter 解析は既存 preflight と同じく `awk` の `^stale: *true *$` 検出で行う (PyYAML 不使用)。

### D3 (#3): Connections セクションを Dataview inline field 化

**Vocabulary** (固定セット):

| key | 意味 |
|-----|------|
| `depends_on` | 依存している (lib / 上流コード / 前提知識) |
| `contradicts` | 矛盾している (両立しない主張・設計) |
| `caused_by` | この理由で生じた (原因) |
| `fixed_in` | ここで修正された (バージョン / commit) |
| `supersedes` | これが古いものを置き換えた |
| `related_to` | その他の弱い関連 (default) |

**Syntax** (Connections セクション内):

```markdown
## Connections

- depends_on:: [[jose]] — JWT library used for Edge runtime auth
- contradicts:: [[old-auth-design]] — old design used cookie sessions
- related_to:: [[edge-runtime]]
```

各行が Dataview inline field として 1 件パースされる。`[[X]] — note` 全体が value になるが、wikilink は graph 上で正しく拾われる。

**後方互換**: `- [[X]] — note` (key なし) は **`related_to::` 同等として読む**。dashboards はこれを許容する DQL で書く (`WHERE contains(string(file.lists), "[[X]]")`)。新規 ingest からは Claude が必ず key を付ける。

**migrate の挙動**: 既存ページの Connections 各行に対し、行頭が `- [[` の場合は `- related_to:: ` プレフィックスを付与する **以上の事はしない** (機械的に意味を推定するのは危険)。

### D4 (#6): MERGE rule の決定木を WIKI.md に明文化

WIKI.md の "MERGE Rule for Entities and Concepts" セクションを以下に置き換える:

```
新 source からの主張 vs 既存ページの記述が衝突した場合、以下の順で判定する:

1. supersession trigger を含む (D2 の linguistic trigger 一覧) →
   旧ページに stale=true + superseded_by を立て、新ページに supersedes を立てる。
   両ページの本文は保持。

2. recency winner: 新主張が直近 30 日以内 AND 既存記述が 90 日以上前
   AND 同じ「現在の状態」に関する事実 (現在使っている lib、現在の設計、etc) →
   新主張を About 段落の冒頭に追加、旧記述に "(過去の記述: 〜 as of YYYY-MM-DD)"
   と prefix を付ける。

3. authority winner: 新 source が以下のいずれか →
   - tags に "paper" / "official" / "spec" を含む
   - source_type が "manual" で domain が空 (= ユーザーが意図的に置いた raw)
   両方 authoritative なら recency に fall through。
   authority winner: 新主張を About 段落冒頭、旧記述は脚注化。

4. else (default): "## Notes (YYYY-MM-DD)" セクションを追加して両論併記。
   confidence は contradictions_present 扱いで 0.5 cap。

判定は単一の sourcing 単位で行い、複数の事実が同時に衝突する場合は事実ごとに 1〜4 を適用する。
```

これにより Claude の judgement に決定的な順序が与えられ、再 ingest での揺れが減る。

### D5: 新 dashboards

`wiki/dashboards/` に 4 ファイル追加:

| ファイル | 用途 | DQL の骨子 |
|---------|------|----------|

### D7: `/wiki-migrate` の v0.9 backfill 内容

idempotent なステップを 3 つ追加。**既存 migrate と同じく PyYAML 等の外部依存は導入しない**。frontmatter は flat scalar のみ (list/dict なし、wikilink リテラルは `[[X]]` の単一値) という前提で、stdlib の Python 3 + 行ベース処理で十分処理できる。

**Step v0.9-A: entity / concept frontmatter backfill**

`wiki/entities/*.md` と `wiki/concepts/*.md` 全件を走査し、frontmatter に以下を追加 (既存があれば再計算で上書き):

```yaml
sources: <D1 の sources_count() の結果>
last_verified: <file.mtime の日付>
confidence: <D1 の式で計算>
```

`stale: true` 付きページは backfill の対象だが、confidence の再計算結果が下がっても上書きする (純関数の一貫性を優先)。stale フラグ自体は backfill では立てない (linguistic trigger 検出は ingest 時のみ実施)。

`contradictions_present` の判定は **D1 で定義した `## Connections` 内の `- contradicts::` 行 grep を完全にそのまま再利用**する (ingest と migrate で同一の純関数)。判定ロジックを片方だけ拡張・修正することは禁止。

**実装: stdlib Python 3 のみで行ベース frontmatter upsert**

`scripts/migrate-entity-confidence.py` 新規。コア処理は以下の擬似コード:

```python
# stdlib only: re, pathlib, sys, datetime, subprocess
# 1. ファイル先頭の "---\n" 〜 "---\n" を frontmatter として抽出
# 2. frontmatter 行を "key: value" 単位で dict に load (flat scalar 前提)
#    - "[[X]]" / "[Y, Z]" / 引用符付き値はそのまま文字列で保持 (yaml load しない)
# 3. sources, last_verified, confidence の 3 キーを upsert
# 4. frontmatter dict を "key: value\n" の連結で再構築 (元の順序を保持、未知キーは末尾に追加)
# 5. body と連結して書き戻し
```

flat scalar 前提が壊れるケース (ユーザーがネスト YAML を手書き) は対象外。事前に `--dry-run` で diff を出してユーザー確認する。検出は「行頭が space で始まる継続行があるか」で判定し、あれば該当ファイルを SKIP + 警告出力。

**Step v0.9-B: Connections 行の `related_to::` プレフィックス化**

各 entity / concept ページの "## Connections" セクション内の各行に対し:

- 行が **正確に** `- [[` で始まる → `- related_to:: ` を行頭に prepend
- 行が `- <key>:: ` (key ∈ vocabulary) で始まる → 触らない
- それ以外 (custom 行など) → 触らない

vocabulary 一致は固定リスト (`depends_on / contradicts / caused_by / fixed_in / supersedes / related_to`) との単純比較。`scripts/migrate-typed-connections.py` 新規 (これも stdlib のみ、awk/sed のレベルで足りる処理だが Python の方が if-then が読みやすいので統一)。

**Step v0.9-C: dashboards コピー**

新規 4 ファイル (low-confidence, stale, contradictions, dependencies) と更新版 README.md を `template/wiki/dashboards/` から `<vault>/wiki/dashboards/` にコピー。**既存ファイルは上書きしない**が、新規追加はする。`cmp -s` で diff チェックして変更があれば追加。

## Risks

### R1: confidence 式が "貧弱" すぎて運用上の意味が薄い

`sources / 5.0` という単純な式は「sources 数だけが効く」モデル。authority weight や recency weight を入れたくなる可能性がある。

緩和: v0.9 では simple-and-deterministic を選ぶ。複雑化は v0.10+ で実データを見てから。confidence は frontmatter なので後で式を変えても再 migrate で全件再計算可能 (純関数)。

### R2: supersession の linguistic trigger が日本語と英語で取りこぼす

handover が日本語で「移行した」と書いていても、Claude が英語パターンでだけ判定すると見逃す。

緩和: D2 の vocabulary に日本語フレーズも明記済み。WIKI.md にも併記。Claude が両言語で判定する。trigger を増やすときは WIKI.md と wiki-ingest.md を両方更新する規約を Notes に書く。

### R3: typed link migrate がユーザー手書きの custom Connection 行を壊す

ユーザーが `- (memo) [[X]]` のような non-standard 形で書いていた場合、`- related_to:: ` を機械的に prepend すると壊れる。

緩和: 行頭が **正確に** `- [[` で始まる行のみ書き換える。それ以外は触らない。dry-run で diff を出してユーザーに確認してもらう。

### R4: dashboards の DQL が `confidence` が無いページで NULL を吐いて見栄えが悪い

migrate 前の vault では confidence が未定義。dashboard が「全部 NULL」を表示する可能性。

緩和: `low-confidence.md` の DQL は `WHERE confidence != null AND confidence < 0.5` で書く (NULL は除外)。migrate 推奨を README にもう一段強調。

### R5: ingest の MERGE 決定木が Claude プロンプト的に長すぎてコンテキスト食う

D4 の決定木を WIKI.md に追加すると、ingest 時の prompt が肥大化する。

緩和: D4 の本文を簡潔に保つ (上記の 4 ステップ提示で十分)。MERGE Rule セクションに局所化、Source Page Format 等とは別配置。WIKI.md 全体は引き続き ingest が一回 Read するだけ。

### R6: 既存 migrate との依存方針整合 (PyYAML を入れない)

現行 `/wiki-migrate` は stdlib (Python 3 標準ライブラリ) のみで動く設計。v0.9 backfill が PyYAML を引き込むと、ユーザー環境で migrate がインストールエラーで落ちる致命傷になる。

緩和: D7 で明記した通り、frontmatter は flat scalar のみ (list/dict なし) という前提で **行ベース処理**に統一。`scripts/migrate-entity-confidence.py` も `scripts/migrate-typed-connections.py` も stdlib only (re, pathlib, sys, datetime, subprocess)。`supersedes: [[X]]` のような wikilink リテラルは引用符無しで読み書きでき、Obsidian / Dataview もそれを受理している。ネスト YAML を手書きしているページは事前検出して SKIP + 警告。

### R7: stale ページの query 除外が「履歴を探したいユーザー」の邪魔になる

ユーザーが「以前 X ライブラリ使ってたよね、なぜやめたんだっけ？」と聞いたとき、stale だからと除外すると答えられない。さらに「X は何に置き換わった？」「X の後継は？」型クエリでは、新ページ Y を読んでも X 側の経緯情報が context に入らないと不正確な回答になる。

緩和: D2 の retrieval rule R-2 (history キーワード) / R-3 (直接ページ参照) / **R-4 (supersession traversal)** の 3 段で機械的にカバー。R-4 は特に重要で、`supersedes: [[X]]` を持つページを include したら X も自動で 1 ホップ追加する。これにより「Y は何を置き換えた？」型のクエリで stale 側 X が確実に context に入る。判定はすべて Bash プリプロセスで決定的に行い、LLM の自然言語判定に揺れない。

## Test plan

### Unit (in-place script test)

1. **confidence 計算**: 仮想 frontmatter (sources=1, no contradictions) で計算 → 0.3。sources=3 → 0.6。sources=5 → 1.0。sources=3 + contradicts:: 行あり → 0.5
2. **supersession trigger 検出**: 「I switched from jose to jsonwebtoken」 / 「jose を jsonwebtoken に置き換えた」 / 「jose is now better」 (これは false) → 各検出結果が期待通り
3. **typed connection parse**: `- depends_on:: [[X]] — note` / `- [[X]] — note` (key 無し) / `- (memo) [[X]]` (custom) → migrate がそれぞれどう書き換えるか確認
4. **YAML 行ベース書き換え**: 既存 frontmatter に `confidence: ...` を idempotent に upsert できるか (2 回実行して diff ゼロ)

### Integration (一時 vault)

5. **fresh vault wiki-init**: v0.9 marker、新 dashboards が配布されること
6. **既存 vault wiki-migrate (dry-run)**: 既存 entity ページに confidence 追加予定であること、Connection が `related_to::` プレフィックス付与予定であることが diff で表示される
7. **既存 vault wiki-migrate (実行)**: 上記が実際に行われること、再実行で diff ゼロ (idempotent)
8. **新規 ingest with supersession**: 「move from jose to jsonwebtoken」を含む handover を ingest → jose ページが stale=true、jsonwebtoken ページに supersedes が立つ
9. **MERGE conflict (recency)**: 古い entity ページに「uses jsonwebtoken」、新 handover に「currently uses jose」(supersession trigger なし) → recency winner で jose が About 冒頭、旧記述に prefix
10. **/wiki-query stale フィルタ R-1 (default)**: stale=true entity を含む vault で `/wiki-query "現在の auth ライブラリ"` → 候補リストに stale ページが含まれず、最終回答も stale 情報を引用しない
11. **/wiki-query R-2 (history keyword)**: 同 vault で `/wiki-query "auth の経緯を教えて"` → stale ページも候補に含まれ、過去の記述が回答に登場
12. **/wiki-query R-3 (direct page reference)**: `/wiki-query "[[old-jose-impl]] について"` → 該当 stale ページが直接 include される
13. **/wiki-query R-4 (supersession traversal)**: `/wiki-query "Y は何を置き換えた？"` で Y (non-stale) ページの `supersedes: [[X]]` 経由で X (stale) も候補入り、置き換え経緯を答えられる
14. **sources_count() 純関数性**: ingest で sources=3 と書かれた entity に migrate を当てる → 同じ計算式で 3 のまま (idempotent)
15. **flat scalar 前提違反検出**: ネスト YAML を手書きした entity ページを vault に置いて migrate 実行 → SKIP + 警告ログが出る、本文は触られない

### Negative
