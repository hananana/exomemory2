# Plan: v0.8.1 — Batched ingest + reliable success detection

## Goal

v0.8.0 の rescue で発覚した 2 つのバグを修正する:

1. **`_is_success` が subagent の result を main の result と誤認**して、正常完了した clip を `clip FAILED rc=0` と判定した。結果として queue 削除されず、`.last-ingest` も進まない。
2. **52 dirty を 1 回の `/wiki-ingest` に渡したことで claude プロセスが事実上 hang**（CPU 6 分 / wall 7 時間）。中途で kill する羽目になり、wiki に何も反映されなかった。

## Non-goals

- 並列 ingest（複数 `claude -p` 同時起動）。`.ingest.lock` を撤回するのは v0.8.0 の整合性を壊すので別案件。
- LLM 側のロングランニング不安定さの根本対処（API リトライ強化など）。バッチ分割で「1 invocation あたりの負荷を下げる」ことで間接的に緩和する。
- preflight 自身の改修。dirty 一覧の生成ロジックは触らず、wiki-ingest 側で head -n で絞る。

## Context

### 観測事実（v0.8.0 リリース直後の実 vault）

`.ingest.log` から:
```
[exomemory2] clip FAILED rc=0 (queue files retained for retry)
[exomemory2] rescue: rebuilt=2 empty=3 err=0 queues=1
[exomemory2] spawned: clip=1 ingest=1 dirty=52 bypass=1
```

- clip フェーズの最終 stream-json result は **`Batch complete CREATED:18 SKIPPED:0 FAILED:2`、`is_error: false`** で**実際は成功している**。
- にもかかわらず `_is_success` は false を返した。
- ログの result レコード集計: メイン session_id `e54251dc` で **20 件**の result。`/exomemory2:wiki-clip --batch` が内部で並列 subagent を起動し、subagent の result も stream-json に流れた。
- 現行 `_is_success` は `grep '"type":"result"' | tail -n 1` で「最後の result」を取るが、subagent の short result が main の Batch complete の **後** に時系列的に追記されるケースがあり、その subagent result の `is_error` で判定してしまう。

ingest 側は `dirty=52` で `/wiki-ingest` を起動 → claude プロセス (PID 28901) が 7 時間 idle 状態で CPU 6 分のみ消費 → `result` レコードを 1 つも書かずに stuck。原因は内部処理の詰まり (大量 subagent / 大量 LLM 呼び出し / プロセスデッドロック等の複合と推測) で、ここで深追いはせず「1 回あたりの負荷を必ず小さく保つ」方向で構造的に回避する。

## Design decisions

### D1: `_is_success` を main session_id に固定する

stream-json の最初の `{"type":"system","subtype":"init",...}` レコードに **その invocation の main session_id** が入る。result レコードは subagent でも main でも同じスキーマで流れるが、`session_id` フィールドで切り分けられる。

```bash
_is_success() {
  local f="$1"
  [ -s "$f" ] || return 1
  # main session_id from the first init record
  local main_sid
  main_sid=$(grep -m 1 '"type":"system","subtype":"init"' "$f" 2>/dev/null \
             | jq -r '.session_id // empty' 2>/dev/null)
  [ -z "$main_sid" ] && return 1
  # last result record whose session_id matches main
  local last_result
  last_result=$(grep '"type":"result"' "$f" 2>/dev/null \
                | jq -c --arg sid "$main_sid" 'select(.session_id == $sid)' 2>/dev/null \
                | tail -n 1)
  [ -z "$last_result" ] && return 1
  local is_error
  is_error=$(printf '%s' "$last_result" | jq -r '.is_error // true' 2>/dev/null)
  [ "$is_error" = "false" ]
}
```

これで subagent の result は判定から除外される。

### D2: `/wiki-ingest` に `--limit N` を導入し、ERROR は LLM 処理から外す

`$DIRTY` には **CREATE / UPDATE のみ** を入れ、`--limit` をそこに適用する。ERROR レコード（slug collision）は LLM 処理を必要としないため Bash 側で件数と slug 一覧を集計し、Step 3 のレポートでまとめて表示する。これで:

- LLM 負荷は **CREATE/UPDATE の N 件で常にキャップ**される（ERROR がいくつあっても 1 invocation の重さに影響しない）
- ERROR が先頭に居座って CREATE/UPDATE が starve する問題も同時に消える（そもそも DIRTY に混入しない）
- ERROR 集計はサマリとして必ず表示されるので、ユーザーは slug collision に気付ける

`commands/wiki-ingest.md` の Step 1 末尾を以下のように改める:

```bash
# DIRTY: only the work the LLM has to do (CREATE/UPDATE).
jq -c 'select(.op == "CREATE" or .op == "UPDATE")' "$MANIFEST" > "$DIRTY"

# Apply --limit to CREATE/UPDATE only. Capping the LLM workload here is the
# whole point: a single invocation must stay small enough to terminate.
if [ -n "${LIMIT:-}" ] && [ "$LIMIT" -gt 0 ]; then
  full=$(wc -l < "$DIRTY")
  head -n "$LIMIT" "$DIRTY" > "$DIRTY.tmp" && mv "$DIRTY.tmp" "$DIRTY"
  kept=$(wc -l < "$DIRTY")
  echo "limited_to=$LIMIT (full_work=$full, deferred=$((full - kept)))"
fi

# ERROR tally (no LLM work; reported in Step 3 from these vars).
ERROR_COUNT=$(jq -c 'select(.op == "ERROR")' "$MANIFEST" | wc -l | awk '{print $1}')
ERROR_SLUGS=$(jq -r 'select(.op == "ERROR") | .slug' "$MANIFEST" | paste -sd, -)
echo "error_count=$ERROR_COUNT"
echo "error_slugs=$ERROR_SLUGS"
```

Step 2 / 3 の wiki-ingest.md 文言修正:

- Step 2 から「For ERROR records (slug collision): do not modify any file. Report to the user.」を **削除**（ERROR は DIRTY に入らないので per-record loop を通らない）
- Step 3 のレポートに「`ERROR: <error_count> (slug collision: <error_slugs>)`」行を **追加**（Bash 出力の `error_count` / `error_slugs` をそのまま使う）

arg parse にも `--limit` を追加。手動実行時 (`/wiki-ingest`) は通常 `--limit` を付けないので **default は無制限**。

### D3: `lib/ingest-spawn.sh` が rescue/auto-ingest 経路で常に `--limit` を付ける

`.exomemory-config` に新キー:

| キー | 既定値 | 意味 |
|---|---|---|
| `INGEST_BATCH_SIZE` | `10` | rescue / auto-ingest が 1 回の `/wiki-ingest` に渡す上限。0 で無効（全件）。 |

`ingest_spawn` の ingest 起動部:

```bash
local ingest_cmd="/exomemory2:wiki-ingest"
if [ "$INGEST_BATCH_SIZE" -gt 0 ]; then
  ingest_cmd="$ingest_cmd --limit $INGEST_BATCH_SIZE"
fi
```

`AUTO_INGEST_INTERVAL_SEC` (既定 1800s) が並走防止になっているので、N 件処理 → 30 分後の SessionStart で再 rescue → 残りの N 件 …… と段階的に進む。

dirty=52 で `INGEST_BATCH_SIZE=10` なら、6 回の SessionStart で完了する見込み。1 回あたり 18 sources で 20 分の前例から、10 sources は **約 10〜15 分 / $5〜8** に収まる想定。

#### D3 補足: rescue 経路の bypass との両立

v0.8.0 の rescue は「dirty が 1 件でも threshold を bypass して即 ingest」する設計だった。これは v0.8.1 でも維持する。BATCH_SIZE は **「1 回あたりの上限」**であり、「最低何件あれば走るか」とは独立。

つまり rescue で 1 件 rebuild → bypass=1 で ingest spawn → wiki-ingest が --limit 10 で 1 件処理して終了、という流れ。整合的。

### D4: バージョン bump と README

`plugin.json` を v0.8.1。README は v0.8 段落に「初回大量取り込みに耐えるバッチ分割を v0.8.1 で導入」を 1 行追記（独立段落は作らない、軽微な改善なので）。

## Risks

### R1: `--limit` の選定順が古いものを後回しにする

現行 preflight は `find` の出力順（≒ inode 順）で dirty を列挙する。`head -n N` で取ると古いセッションが先か新しいセッションが先かは vault の状態次第で予測しづらい。

評価: 全件いずれは取り込まれるので、順序は機能性に影響しない。ただし「ユーザーが直近の会話を早く wiki に反映したい」ケースで遅れるリスクはある。

緩和: 現バージョンでは順序ソートしない。次の v0.9 で「mtime 新しい順」を入れる余地を残す（v0.8.1 のスコープ外）。

### R2: 部分 ingest 中にユーザーが手動 `/wiki-ingest` する競合

v0.8.0 の `.ingest.lock` は **ingest_spawn 経路の中だけ**で取得・解放される。手動 `/wiki-ingest` は lock を取らない。同時に走ると wiki/sources/ への書き込みが衝突しうる。

評価: これは v0.8.0 から既存のリスクで、本 PR で増えない。本 PR では明示的に対処しない（別途 v0.9 検討）。

### R3: `INGEST_BATCH_SIZE` のデフォルト値の妥当性

10 は「前回 18 sources / 20 分」から逆算した安全側の値。実環境で 10 件でも stuck が起きた場合は、ユーザーが `.exomemory-config` で 5 とかに下げられる。0（無制限）にも戻せる。

### R4: `_is_success` で `init` レコードが見つからない場合

`grep -m 1 init` が空を返したら現状の修正案では `return 1`（FAILED 扱い）。これは安全側に倒す挙動で正しい（ログが壊れていれば conservative に失敗扱い）。

### R5: subagent の result が main の result より後に流れる前提

D1 の修正は「main session_id の result だけ見る」ので順序に依存しない。仮に subagent の result が main より早く流れても、後から流れても、main の result だけを `tail -n 1` で取る。

### R6: ERROR レコードによる starvation と LLM 負荷膨張

問題は 2 つ複合する:

1. `--limit N` を dirty 全体に適用すると、ERROR が先頭に居座った場合に CREATE/UPDATE が永遠に到達しない (preflight が ERROR を毎回再検出するため)
2. ERROR を毎回全件 LLM に渡すと、ERROR が大量に蓄積した vault では batch 化の意味が崩れる

D2 で **ERROR を DIRTY (LLM 処理対象) から完全に外し、Bash 側で件数 + slug 集計のみ Step 3 に渡す** ことで両方解決する。LLM 負荷は CREATE/UPDATE の N 件で固定キャップ、ERROR は集計されるだけなので invocation の重さには影響しない。starvation も発生しない。

## Test plan

### Unit-ish

1. `_is_success` の **subagent result 混入テスト**: 手作りの stream-json ファイル（init + main result is_error:false + subagent result is_error:true の順）を渡して、_is_success が 0 (success) を返すことを確認
2. `_is_success` の **main result is_error:true テスト**: main result が is_error:true なら 1 (failure) を返す
3. `_is_success` の **init レコード欠損テスト**: 1 (failure) を返す
4. wiki-ingest の `--limit` 解析テスト: dirty=20 で `--limit 5` を渡すと、Step 2 で処理されるのが先頭 5 件のみ、`limited_to=5 (full_dirty=20, deferred=15)` のログが出る

### Integration

5. 一時 vault で 15 件の dirty handover を作り、`INGEST_BATCH_SIZE=5` で ingest_spawn を呼ぶ → 5 件のみ処理される
6. 既存 capture 経路の non-regression: 普通の SessionEnd で `INGEST_BATCH_SIZE` が効くこと（あるいは 0 設定で従来通り全件処理されること）

## Rollout

1. lib/ingest-spawn.sh と commands/wiki-ingest.md の改修
