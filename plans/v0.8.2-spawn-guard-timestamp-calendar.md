# Plan: v0.8.2 — spawn guard + handover timestamp + calendar layout cap

3 件のバグ / UX 問題をまとめて v0.8.2 に入れる。いずれも v0.8.0 の rescue 出荷後に実 vault で発覚した症状の対処。

## Goals

1. **B1 spawn guard**: `bypass_threshold=1` でも `DIRTY=0` のときは ingest を spawn しない（無駄な `claude -p` 起動を抑制）
2. **B2 handover timestamp**: rebuild 時に `last_captured_at` を「rebuild 時刻」ではなく **transcript の最初メッセージ時刻** から取る（rescue で大量 rebuild が同日に集中する根本原因の対処）
3. **B3 calendar layout cap**: Obsidian Dataview CALENDAR の day cell が大量 dot で隣行を押し出す問題を CSS snippet で抑える

## Non-goals

- Calendar の正確な「過密日件数」表示（heatmap で代替できるので追加しない）
- 既存の rebuild 済み handover の captured_at を **強制的に** 後追い修正するスクリプト（次回 rescue 時に mtime 比較で自然に取り直されるので不要）
- Dataview CALENDAR の根本仕様改修（Obsidian community plugin に手は出さない）

## Context

### B1 の根拠

v0.8.1 のログで観測:
```
[exomemory2] rescue: rebuilt=0 empty=3 err=0 queues=1
[exomemory2] spawned: clip=1 ingest=1 dirty=0 bypass=1
```

`should_ingest` 判定が `bypass_threshold=1` の場合に `DIRTY` を見ていないため、dirty=0 でも `should_ingest=1` で spawn される。clip 経路のために spawn が必要なケースもあるので spawn 自体は問題ないが、**ingest 側の `claude -p` が無条件で起動するのは無駄なコスト**。

### B2 の根拠

Calendar 画像で 4/26 にドット 26 個が集中していた件、原因は v0.8.0 rescue で 26 件 rebuild した結果、すべての handover の `last_captured_at` が rebuild 実行時刻 (4/26 深夜) で書かれたこと。`handover-build.sh:67` の `captured_at="$(date -u '+%Y-%m-%dT%H:%M:%SZ')"` がこの瞬間時刻を入れる設計。

本来は **そのセッションの最初の発話時刻** を入れるのが意味的に正しい。Calendar の軸も heatmap の軸も「会話が起きた日」が知りたいのであって「handover が物理的に書かれた日」ではない。

### B3 の根拠

Dataview CALENDAR (Svelte コンポーネント、scoped class `.svelte-q3wqg9` 系) は 1 日のページ数だけドットを並べる仕様で、件数上限がない。dot 数がセル幅を超えると複数行に流れ、`min-height: 100%` で隣行を押し出してレイアウト崩壊。`.day` cell に CSS を当てれば抑制できる（Svelte hash と同階層の base class なので `.day` 単体で selector 可能）。

## Design decisions

### D1 (B1): `should_ingest` に `DIRTY > 0` を必須化

`lib/ingest-spawn.sh` の判定を:

```diff
-    if [ "$threshold_ok" = "1" ] && [ "$interval_ok" = "1" ]; then
+    if [ "$DIRTY" -gt 0 ] && [ "$threshold_ok" = "1" ] && [ "$interval_ok" = "1" ]; then
       should_ingest=1
     fi
```

`bypass_threshold=1` は **「最低件数の閾値を bypass」する意味**であり、「dirty=0 でも走る」とは違う。dirty=0 なら ingest 側で何もすることがないので spawn 不要。clip 側 (has_queue) は別ゲートなので影響なし。

### D2 (B2): `build_handover` が transcript の first message timestamp を採用 + `captured_at_source` フィールドを書く

`lib/handover-build.sh` の `captured_at` 取得を:

```diff
-  local captured_at
-  captured_at="$(date -u '+%Y-%m-%dT%H:%M:%SZ')"
+  # Prefer the timestamp of the first user/assistant message in the transcript,
+  # so calendar/heatmap reflect when the conversation actually happened, not
+  # when this handover happens to be (re)built. Fall back to "now" only when
+  # the transcript has no usable timestamp (older Claude Code versions).
+  local captured_at captured_at_source
+  captured_at="$(
+    jq -r '
+      select((.type == "user" or .type == "assistant") and (.timestamp != null))
+      | .timestamp
+    ' "$transcript_path" 2>/dev/null | head -n 1
+  )"
+  captured_at="$(printf '%s' "$captured_at" | sed -E 's/\.[0-9]+Z?$/Z/')"
+  if [ -n "$captured_at" ]; then
+    captured_at_source="transcript-first-message"
+  else
+    captured_at="$(date -u '+%Y-%m-%dT%H:%M:%SZ')"
+    captured_at_source="fallback-now"
+  fi
```

frontmatter にも新フィールドを書き出す:

```diff
   {
     printf -- '---\n'
     printf 'title: "Session %s"\n' "$session_id"
     printf 'session_id: "%s"\n' "$session_id"
     printf 'last_trigger: "%s"\n' "$trigger"
     printf 'last_captured_at: "%s"\n' "$captured_at"
+    printf 'captured_at_source: "%s"\n' "$captured_at_source"
     printf -- '---\n\n'
     printf '%s\n' "$body"
   } > "$tmp_md" || ...
```

#### 既存 handover への波及（修正版）

rescue-orphans.sh での自動強制 rebuild は **採用しない**（`last_trigger` を識別子に使うと、rebuild 後も同じ trigger が書かれて永久ループになるため）。代わりに **`/wiki-migrate` の責務**として、`captured_at_source` フィールドが `transcript-first-message` で **ない** handover を一括再 rebuild する。

`commands/wiki-migrate.md` に新ステップを追加（idempotent）:

```bash
# Step N: backfill last_captured_at from transcript timestamp for any handover
# that lacks captured_at_source: transcript-first-message (i.e. pre-v0.8.2,
# or built by an older rescue path before B2 landed).
SCRIPT_DIR="${CLAUDE_PLUGIN_ROOT}"
source "$SCRIPT_DIR/lib/handover-build.sh"

migrated=0
skipped=0
for f in "$VAULT/raw/handovers/"*.md; do
  [ -f "$f" ] || continue
  if grep -q '^captured_at_source: "transcript-first-message"$' "$f"; then
    skipped=$((skipped + 1))
    continue
  fi
  sid="$(basename "$f" .md)"
  transcript="$(find "$HOME/.claude/projects" -maxdepth 4 -name "${sid}.jsonl" -not -path '*/subagents/*' -print -quit 2>/dev/null)"
  if [ -z "$transcript" ]; then
    echo "  SKIP: $sid (transcript not found)"
    continue
  fi
  if build_handover "$VAULT" "$transcript" "$sid" "wiki-migrate-backfill"; then
    migrated=$((migrated + 1))
  fi
done
echo "captured_at backfill: migrated=$migrated, already-current=$skipped"
```

これは:
- v0.8.2 リリース直後にユーザーが `/wiki-migrate` を 1 回実行すると、過去 73 件の rescue 由来 handover が再 rebuild される（captured_at が本来の会話時刻に戻る）
- その後の `/wiki-migrate` は `captured_at_source: "transcript-first-message"` で全件スキップ → idempotent
- rescue-orphans.sh は **触らない**（D2 #3 のオリジナル mtime 比較ロジックのまま）→ 永久ループのリスクなし
- 追加の `/wiki-migrate` 実行 = ユーザー明示操作なので暴発しない

#### transcript timestamp 不在の idempotency 担保

migrate は **transcript timestamp が取れるとき限定**で build_handover を呼ぶ。timestamp が無い transcript に対しては書き換えず、handover を放置する:

```bash
ts=$(jq -r '... .timestamp' "$transcript" | head -n1)
if [ -z "$ts" ]; then
  skipped_no_timestamp=$((skipped_no_timestamp + 1))
  continue
fi
build_handover ...
```

これで:
- 通常の transcript (timestamp あり) → migrate で 1 回 backfill → 以後 `captured_at_source: "transcript-first-message"` でスキップ
- 旧 transcript (timestamp なし) → migrate がスキップ → 既存の handover はそのまま
  - その handover の captured_at_source は `"fallback-now"` または不在のまま
  - migrate を何度実行しても触られない
  - 副作用なし

`build_handover` 自身は捕捉経路 (PreCompact/SessionEnd) では timestamp 不在時に従来通り `date -u` の fallback を使う（capture.sh は migrate と違って「いま書くしかない」ので fallback が必要）。fallback 経由で書かれた handover に `captured_at_source: "fallback-now"` が残っても hash の安定性は保たれる（same input → same output、preflight は SKIP）。

### D3 (B3): CSS snippet による day cell 高さ固定 + dot overflow:hidden

新規ファイル `template/snippets/exomemory2-calendar.css`:

```css
/* exomemory2 — keep Handover calendar from blowing up row height.
 *
 * Dataview's CALENDAR (Svelte, scoped class .svelte-XXXXXXX) renders one
 * dot per page on each day cell with no built-in cap. A day with dozens of
 * handovers — typical right after a rescue rebuild — overflows the cell and
 * pushes the next week's row down, breaking the grid layout.
 *
 * We bound .day cells uniformly and clip the dot row. Exact daily counts
 * are still visible via the v0.5 Activity heatmap above the calendar.
 */

.markdown-rendered .day {
  min-height: 4em;
  max-height: 5em;
  overflow: hidden;
}

.markdown-rendered .day > svg,
.markdown-rendered .day .dots {
  max-height: 2em;
  overflow: hidden;
}
```

#### 配置と有効化

- 新規 vault: `wiki-init` が `<vault>/.obsidian/snippets/exomemory2-calendar.css` を配置 + `appearance.json` の `enabledCssSnippets` に追加（jq で merge）
- 既存 vault: `wiki-migrate` が同じ操作を実行。idempotent (既に存在すれば上書きしない / `enabledCssSnippets` に重複追加しない)

`appearance.json` のマージ:
```bash
jq --arg s "exomemory2-calendar" \
  '.enabledCssSnippets = ((.enabledCssSnippets // []) + [$s] | unique)' \
  "$VAULT/.obsidian/appearance.json" > tmp && mv tmp "$VAULT/.obsidian/appearance.json"
```

#### 効かなかった場合のフォールバック

CSS が効くかは Obsidian + Dataview のバージョンに依存する。動作確認は **ユーザーが Obsidian で実視認する**しかない。効かない場合に備えて:

- snippet ファイルに「class 名が違うバージョンに当たったら下記 selector を試せ」のコメントを書いておく
- 万一動かなければ DevTools で inspect → 正しい selector に差し替え（v0.8.3 以降）

## Risks

### R1: B2 の transcript timestamp が想定と違う形式

Claude Code のバージョンによって transcript の `.timestamp` フィールドが存在しない / 形式が違う可能性。

緩和: jq で取れなければ `date -u` の fallback、形式 normalization で fractional seconds を削る、ISO 8601 broad に対応。

### R2: wiki-migrate での backfill 大量 UPDATE のコスト

ユーザーが `/wiki-migrate` を実行した直後、過去 73 件の rescue 由来 handover が再 rebuild される。`build_handover` 自体は jq だけのローカル処理でコストほぼゼロだが、その後の自動 `wiki-ingest` で 73 件すべてが UPDATE 判定（hash 変わる）→ 73 件分の LLM ingest 発生。

緩和: v0.8.1 の `INGEST_BATCH_SIZE=10` がそのまま効く。10 件 / SessionStart × ~7 回 で完了（正常運用）。さらに、migrate 直後は ingest が 1 回 spawn されるだけ（auto-ingest gate 経由）なので、ユーザーが意図したタイミングでだけ発火する。

### R3: B3 CSS snippet が Dataview バージョンで効かない

`.day` という汎用的すぎる selector を使うため、Obsidian の他コンポーネントにも影響する可能性。

緩和: selector を `.markdown-rendered .day` で限定（preview/render mode 内のみ）。それでも誤命中するなら `.dataview .day` のように更に限定する v0.8.3 修正の余地。

### R4: `appearance.json` マージで既存設定が壊れる

ユーザーが手で書いた他の snippets / theme 設定との衝突。

緩和: jq で `enabledCssSnippets` フィールドだけ touch、他フィールドは保持。idempotent (`unique` で重複追加しない)。バックアップとして migrate 前に `appearance.json.bak` を作る。

### R5: snippet が "enabled" 設定だけでは効かない（Obsidian の手動有効化が必要）

`appearance.json` の `enabledCssSnippets` に追加しても、Obsidian は再起動 / Settings で snippet を refresh しないと反映しないことがある。

緩和: `wiki-init` / `wiki-migrate` の最後に「Obsidian を一度開き直すと snippet が有効になる」と明示出力する。

## Test plan

### Unit

1. **B1**: ingest_spawn を AUTO_INGEST=1 / dirty=0 / queue 空 / bypass=1 で呼び spawn が echo されないこと
2. **B1**: 同条件 + queue ありのとき、clip だけ spawn される（ingest は spawn されない）
3. **B2**: transcript に `{"timestamp":"2026-04-12T10:00:00.123Z",...}` を含む handover を build → captured_at が `2026-04-12T10:00:00Z` に揃うこと

5. **D2 idempotency**: 一時 vault に `captured_at_source: "transcript-first-message"` 既存 handover を置いて wiki-migrate を 2 回連続実行 → 2 回目で全件スキップ、ファイル mtime 不変であること
6. **D2 transcript timestamp 不在**: timestamp フィールドが無い transcript 由来の handover に対して migrate が **何度実行しても触らない** こと

### Integration

7. **B3**: 一時 vault で wiki-init → snippet ファイルが配置 + appearance.json に追加されることを確認
8. **B3**: 既存 vault に wiki-migrate を当てる → 同様に配置されること、idempotent (二度実行しても重複しない)
