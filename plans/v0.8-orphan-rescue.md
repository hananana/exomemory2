# Plan: v0.8 — SessionStart orphan rescue

## Goal

`prefix + x` などで Claude Code セッションが SIGHUP 即死した場合でも、その transcript が次回以降のセッションで自動的に handover 化 + ingest される構造を導入する。

副次的に、これまで取りこぼされた**過去 30 件相当の孤児セッション**を同じロジックで回収する（個別の救済スクリプトを書かない）。

## Non-goals

- tmux 側の graceful kill 実装（v0.8 で構造的に解決するので不要と判断、議事録参照）。
- handover 化の質的向上（要約・タグ付け等）。本 PR は「捕捉漏れをゼロにする」だけにスコープを絞る。
- transcript の取りこぼし監視ダッシュボード化。`log.md` 既存の流れに乗せるのみ。

## Context

### 取りこぼしの実態（2026-04-26 計測）

| 指標 | 値 |
|---|---|
| 全 transcript JSONL（subagents 含む） | 224 |
| メインセッションのみ | 80 |
| メインセッション かつ 2026-04-18 以降 | 75 |
| うち handover 未生成（**真の孤児**） | **30** |
| 孤児 transcript の総サイズ | 37 MB |
| `.clip-queue/` 孤児ファイル | 1 ファイル / 20 URL |

`subagents/agent-*.jsonl` は Claude Code 内部のサブエージェント実行ログであり、handover 化対象外。`find ... -not -path '*/subagents/*'` で除外する。

### 原因

`tmux prefix + x`（`kill-pane`）は対象ペインの foreground プロセスグループに SIGHUP を送るため、Claude Code は SessionEnd hook を発火させる暇なく死ぬ。`capture.sh` も `nohup claude -p ... &`（auto-ingest）も走らない。したがって:

- `raw/handovers/<session_id>.md` が作られない
- `.clip-queue/<session_id>.txt` に積まれた WebFetch URL が孤児化（session_id ベースで分離されているため別セッションが拾わない）

### 救済の方向性

「SessionEnd 経路の信頼性を上げる」のではなく、「**SessionStart 起動時に未処理の transcript を自動発見して処理する**」設計に倒す。これにより:

- tmux の閉じ方を問わない（kill / クラッシュ / ターミナル落ち全部に効く）
- 過去分の救済は「初回 SessionStart で 30 件ぶんが拾われる」という形で自動的に完了する
- 構造的に 1 経路にまとまるので、tmux graceful kill のような副次的対策は不要

## Design decisions

### D1: SessionStart レスキュー機構を導入し、tmux 側対策は採用しない

`session-start.sh` を拡張し、未 handover 化の transcript を発見してバックグラウンドで `capture.sh` 相当の処理を流す。

採用しない選択肢:
- `tmux.conf` の `prefix + x` rebind: hook 発火に最大数十秒かかる UX 劣化、`run-shell -b` 必須、claude 以外のプロセスでも誤判定リスク。**v0.8 が入れば不要**。
- 周期的 cron: SessionStart のほうがイベントドリブンで設計が単純。

### D2: 対象範囲のフィルタ

レスキュー対象は次の **AND** 条件すべてを満たす transcript のみ:

1. `~/.claude/projects/*/[^/]*.jsonl`（メインセッション、subagents 除外）
2. mtime が `--since` 期間内（既定 **14 日**、リポジトリ作成 2026-04-18 をカバー）
3. **下記いずれか**を満たす:
   a. 同名 `<session_id>.md` が `<vault>/raw/handovers/` に**存在しない**
   b. transcript の mtime が同名 handover の mtime より**新しい**（PreCompact で生成された中間 handover がある「途中まで処理済みセッション」を救うため）
4. `<session_id>` が**現セッション自身ではない**（hook 入力 JSON の `session_id` を除外）
5. 該当 transcript ファイルが**直近 N 秒以内に書き込まれていない**（既定 60s。生きているセッションを誤って捕捉しないため）

#### D2 #3 の根拠

現行 `capture.sh` は `PreCompact` でも `SessionEnd` でも同じ `<session_id>.md` を**上書き**する設計（`hooks.json` で両 hook が同じスクリプトに紐づく）。つまり「コンテキスト圧縮を 1 度経験したあと `prefix + x` で死んだセッション」は handover ファイルが存在するため、`exists` 判定だけだと **最終 SessionEnd 相当の更新を永久に取りこぼす**。

mtime 比較を入れることで、`raw/handovers/<sid>.md` が古ければ最新 transcript で再生成される。 `handover-build.sh` は冪等なので再実行コストはほぼゼロ。

### D3: `capture.sh` から共通ロジックを切り出す

`capture.sh` は現在「stdin から hook 入力 JSON → transcript_path / session_id を抽出 → handover 書き出し → ingest spawn」の単一経路。これを以下に分解する:

- `lib/handover-build.sh`（新規・source される）: `transcript_path / session_id / trigger / vault → handover.md` の純関数
- `lib/ingest-spawn.sh`（新規・source される）: `vault / has_queue / should_ingest → background spawn`。lock / interval / dirty ゲートも含む
- `capture.sh`: 上記 2 つを呼ぶ薄い hook（既存挙動と互換）
- `session-start.sh`: 既存の wiki freshness 通知 + **新設レスキューパス**（後述）

切り出しの目的は単一責任化と、レスキュー側からの再利用。SessionEnd / SessionStart 双方が同じロジックを通る。

### D4: SessionStart レスキューはバックグラウンドで spawn

`session-start.sh` の hook タイムアウトは 5s（hooks.json）。同期で 30 件処理は不可能。次の二段で動かす:

1. **同期パート**（〜数百 ms）: 既存の wiki freshness 通知のみ JSON で返す（ユーザー体験を変えない）
2. **非同期パート**: `( rescue_orphans ... ) </dev/null >/dev/null 2>&1 & disown`
   - `find` で候補列挙 → D2 のフィルタ通過分だけ `lib/handover-build.sh` で順次 md 化
   - 全件分の md 化が終わったら、既存 `lib/ingest-spawn.sh` を呼ぶ（dirty 数 / interval ゲートはそのまま尊重）

handover 化（jq 抽出）は LLM 呼び出しを伴わないので、30 件でも数秒で終わる想定。ingest だけが LLM 起動。

### D5: rescue 経路は **threshold を bypass**、lock / interval / 成否判定は再利用

レスキュー経路の ingest ゲートは平常時 (`capture.sh`) と意図的に **異なる**:

| ゲート | 平常時 (SessionEnd) | rescue 時 (SessionStart) |
|---|---|---|
| `.ingest.lock` (排他) | 尊重 | **尊重** |
| `AUTO_INGEST_INTERVAL_SEC` (連投防止) | 尊重 | **尊重** |
| `AUTO_INGEST_THRESHOLD` (dirty 件数下限) | 尊重 (既定 3) | **bypass する条件あり** ↓ |

**threshold bypass ルール**: rescue が当該起動で **handover を新規生成または上書きした件数 ≥ 1**、または **救済対象 queue ファイルが ≥ 1** のとき、`AUTO_INGEST_THRESHOLD` を無視して ingest を spawn する。

bypass しない場合の問題: 「`prefix+x` で 1 セッションだけ孤児化、queue なし」という最も典型的なケースで dirty=1 < threshold=3 となり、handover 再生成だけで ingest が動かず救済が完結しない。これでは v0.8 のゴール「自動的に handover 化 + ingest」が満たされない。

bypass しない場合（rescue が何も新規 / 更新を作らなかった）は、無駄な ingest を避けるためそのまま既存ゲートに従う（実質スキップ）。

interval ゲートは**尊重したまま**: rescue 直前に正常な ingest が走った場合（30 分以内）はスキップし、自然に次回 SessionStart で再試行されるのに任せる。短時間連投を避けるための保護はそのまま残す。

短時間に複数セッションが起動して同時に rescue を spawn する事故は、既存の `<vault>/.ingest.lock` (noclobber + PID liveness) がそのまま防ぐ。

----

加えて**現状の `capture.sh` には別の穴があり、本 PR で塞ぐ**:

> 現在: `claude -p` の終了コードを問わず、`should_ingest=1` なら `.last-ingest` を無条件更新し、queue ファイルも無条件削除する（capture.sh:329-336 の trap）。

→ ingest が API エラー等で失敗しても `.last-ingest` だけが進み、queue は消える。**取りこぼした URL が永久に消失するデータロス経路**。

本 PR の `lib/ingest-spawn.sh` では:

- `claude -p` の **終了コード**を取得し、加えて stream-json output の最終 `result` レコードの `is_error` を確認する
- **両方が成功** のときのみ `.last-ingest` 更新 + queue 削除
- 失敗時は queue を残し、`.ingest.log` に明示的にエラー行を記録（既存の log を活かす）
- 既存の handover md は dirty 判定で次回再試行される（ingest preflight が hash で UPDATE 検出する）

これは v0.8 の付随改修として明示する。**capture.sh の既存挙動を変更する破壊的修正だが、データロス経路を残すよりは安全**と判断。

さらに、レスキュー固有の `RESCUE_*` 系設定だけ新たに `.exomemory-config` に足す（D7）。

### D6: `.clip-queue/` 孤児ファイルの救済

孤児 queue ファイル（`<session_id>.txt` だが該当 session_id がもう動いてない）も同じレスキューで拾う。**mtime ベースの寝かせ条件は使わない**（次セッション起動時に即時救済するのが目的なので 1 日待つのは矛盾する）。

候補抽出ルール:
- `<vault>/.clip-queue/*.txt` のうち、ファイル名 `<session_id>.txt` の `<session_id>` が **現セッション自身ではない** すべて
- 加えて quiescence ガード: 「該当 `<session_id>` の transcript が D2 #5 と同じ条件で生きていない」ことを念のため確認（マルチターミナル運用で他の生きてるセッションの queue を奪わないため）。`lsof -t -- <transcript>` で現役判定、または mtime > `RESCUE_QUIESCENCE_SEC` で fallback

各ファイルにつき `claude -p '/exomemory2:wiki-clip --batch <file> --captured-by auto-rescue'` を**通常の ingest spawn 内に追加**で投げる。成功時のみ当該 queue ファイルを削除（D5 改訂版に合わせる）。

実装上は `lib/ingest-spawn.sh` に「orphan queue files 引数」を渡せるようにする。SessionEnd 経路では既存どおり自セッションの queue だけ、SessionStart レスキュー経路では孤児 queue 全部を渡す、という違いを引数で切り替える。

### D7: `.exomemory-config` に新規キー

| キー | 既定値 | 意味 |
|---|---|---|
| `RESCUE_ORPHANS` | `1` | 0 で全機能オフ（脱出弁） |
| `RESCUE_SINCE_DAYS` | `14` | mtime フィルタ |
| `RESCUE_QUIESCENCE_SEC` | `60` | 「生きてるセッション」と判定する mtime 経過秒の閾値 |

`.exomemory-config` の strict parser は現状 `KEY=INT` のみ受理する設計なので、すべて整数で表現できる形に揃える。

### D8: バージョン bump

`plugins/exomemory2/plugin.json` を v0.8.0 に。`README.md` のリリース欄に「SessionStart orphan rescue」を 1 行追記。

## Risks

### R1: 生きているセッションの transcript を誤捕捉

mtime ベースの quiescence 判定（既定 60s）は heuristic。ユーザーが思考時間で 60s 以上タイプしていない瞬間に、別セッションのレスキューが走ると、その transcript を「孤児」と誤認する可能性がある。

緩和:
- 自セッションは hook 入力 JSON の `session_id` で確定的に除外（D2 #4）
- 他のメインセッションの PID を `lsof -t -- <jsonl>` で取得し、生きていれば除外する追加ガード（実装容易）。これで mtime ガードは fallback に降格

### R2: 同時 SessionStart の競合

複数ターミナルでほぼ同時にセッションを開いた場合、両方がレスキューを spawn する。handover_build は `<vault>/raw/handovers/<session_id>.md` への上書きで決着するが、ingest は既存の `.ingest.lock` で 1 つに収束する（既存機構の流用）。

ただし `lib/handover-build.sh` 自体が同じファイルへの並列書き込みになると content corruption の懸念。対策: 出力先 md を `<session_id>.md.tmp.<pid>` に書いて `mv` で原子置換する。

### R3: 古い transcript の混入

リポジトリ作成 (2026-04-18) より前の transcript（exomemory2 と無関係なプロジェクトでの会話）まで巻き込む可能性。

緩和:
- D2 #2 の `--since 14d` で時間窓を切る（リポジトリ作成 8 日前の今、過去はカバー、より過去はデフォルトでは出ない）
- `.exomemory-config` の `RESCUE_SINCE_DAYS` でユーザーが調整可

### R4: handover 化が成功するが ingest だけ失敗するケース

handover_build は jq だけなのでほぼ失敗しない。ingest は claude CLI 必須・コスト発生。CLI 不在 / API 障害で ingest 側だけ失敗する可能性がある。

**重要**: 現行 `capture.sh` は失敗してもデータロスする経路を持つ。D5 改訂版で塞ぐ前提:

- handover md は dirty 判定の hash 比較で次回 ingest 時に自動 UPDATE される（既存 preflight ロジック）
- `.last-ingest` と `.clip-queue/*.txt` は**ingest が成功した場合のみ**進める（D5 で挙動変更）

これにより API 障害発生時でも、handover md と queue の両方が次回 SessionStart で再試行可能な状態として残る。

### R5: `lsof` 依存

R1 緩和で `lsof` を使う場合、macOS にはデフォルトで入っているが、念のため `command -v lsof` で fallback を用意する。`lsof` 不在時は mtime ガードのみで判定。

### R6: SessionStart hook タイムアウト

5s 制限内で必ず return すること。同期パートは freshness 通知の JSON 出力までで完結し、レスキュー本体は disown 済みバックグラウンドで動く設計（D4）。

### R7: 過去 30 件の一気 ingest コスト

30 件 / 37 MB を一度に ingest すると LLM コスト急増のリスク。前回 18 sources で $11、単純比例で $18-20 程度の見込みだが、複数セッション分の dedup（hash 一致 SKIP）で実数はさらに減る想定。

緩和: `RESCUE_SINCE_DAYS` を初回 7 日で動かして様子見し、その後 14 → 30 と段階的に広げる運用を README に書く。

## Test plan

### Unit-ish

1. **`handover-build.sh` の冪等性**: 同じ transcript で 2 回呼んでも md 内容が一致する（mv による原子置換確認）
2. **空セッション skip**: tool_use のみの transcript を渡して md が生成されないことを確認（既存 capture.sh の仕様維持）
3. **rescue-orphans.sh の対象選定**: mock で `~/.claude/projects/` を作り
   - subagents 配下が除外される
   - `--since` 期間外が除外される
   - 既存 handover ありが除外される
   - 自セッション (`session_id` 引数) が除外される
   - quiescence 期間内 (mtime fresh) が除外される
   それぞれ単独で確認

### Integration

4. **SessionStart レスキュー dry-run**: `RESCUE_ORPHANS=1` の状態で新セッションを開き、`.ingest.log` にレスキュー由来の `claude -p /exomemory2:wiki-ingest` invocation が記録されること
5. **過去 30 件の救済**: 本実装をマージ後、最初の SessionStart で `raw/handovers/` に 30 ファイル前後が追加されることを確認（`ls | wc -l` で 46 → 76 程度に）
6. **`.clip-queue/73772bd8-...txt` の救済**: SessionStart 後に該当ファイルが削除され、`raw/web/` に 20 件のクリップが生成されることを確認
7. **既存 SessionEnd 経路の non-regression**: `prefix + x` ではなく `/exit` で閉じた場合、従来どおり当該セッション末尾で ingest が spawn されること

### Negative

8. **`RESCUE_ORPHANS=0`**: フラグ off で全機能が走らないこと（`.ingest.log` に新規エントリなし）
9. **vault 未設定**: `EXOMEMORY_VAULT` 未 export 時に rescue が silent skip すること（既存 capture.sh と同じガード）
