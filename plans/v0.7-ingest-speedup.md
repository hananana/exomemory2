# Plan: v0.7 — `/wiki-ingest` Speed Overhaul

> **Status: Layer 1 + 2 + hook preflight が shipped。Layer 3 (並列 subagent) は LLM compliance で実測 fail、撤回。Layer 4 (batch finalize + update-source.sh) は v0.8+ に deferred。詳細は末尾の「実装結果」参照。**


## Goal

`/wiki-ingest` を「とにかく速く」する。1ms でも削る。
現状は `claude -p` 子プロセスが 35+ 個の raw ファイルを順番に LLM で processing しており、`.ingest.log` が一回 4.4MB（vault: ~/vault, 35 handovers + 20 web clips, sources=50）。
SKIP しか発生しない hot path でも数分かかっている疑いがある（実測未取得、今プランの最初のタスクで取る）。

## Non-goals

- WIKI.md スキーマ変更なし（v0.4 schema marker 据え置き）
- frontmatter フィールド追加・削除なし
- ingest の出力（生成される source/entity/concept page の文面・構造）の互換性は維持。前回 ingest の出力と diff が出ない冪等性を保つ
- 並列 subagent 化のために subagent SDK / プラグイン依存を増やさない（Claude Code の Task tool で完結）
- `/wiki-clip`, `/wiki-query`, `/wiki-migrate`, `/wiki-gc` の挙動変更なし
- WebFetch auto-clip / capture hook の挙動変更なし
- `wiki/dashboards/` への影響なし

## ボトルネック仮説（実測前）

現 `/wiki-ingest` 1 回あたりの主なコスト:

1. **SKIP 判定が LLM 主導** — 35 ファイル全てに対して、LLM が「`wiki/sources/<slug>.md` を Read → frontmatter から `source_hash` 抽出 → shasum 比較 → log.md に SKIP 行を Edit append」を直列で走らせる。SKIP が大半なので最大の無駄
2. **WIKI.md 16KB を毎回 Read** — schema は基本不変、毎回読む必要なし
3. **Vault 解決で bash 3 回** — 1 回にまとめられる
4. **dirty ファイルの extract が直列** — N dirty なら N×T 秒。subagent で並列化すれば ~T に
5. **bookkeeping (index/overview/log) が dirty ファイル毎に Edit** — 1 ingest 内で同じファイルに何度も Edit が走る
6. **dirty ファイルが大きい** — 最大 140KB の handover を main context に取り込むので token も時間も食う

## アプローチ

「LLM がやる必要のない仕事を全部 Bash に押し出し、残った LLM 仕事は並列化」の方針で 4 層に分ける:

### Layer 1: Bash Preflight（最大の効き）

`plugins/exomemory2/scripts/ingest-preflight.sh` を新設。引数: `<VAULT> [<RAW_TARGET>]`。stdout に NDJSON 出力、stderr に進捗。

各 raw ファイルについて以下を pure Bash + awk + shasum で計算:

- `path` (絶対)
- `source_id`, `slug`, `source_type` (path から決定論的)
- `source_hash` (`shasum -a 256`)
- `is_empty` (frontmatter strip 後に non-whitespace があるか)
- `existing_source_id`, `existing_source_hash` (wiki page があれば awk で frontmatter parse)
- `op`: CREATE / UPDATE / SKIP / SKIP-empty / ERROR (slug collision)
- `word_count`, `reading_time_min`
- handover の場合: `session_id`
- web-clip の場合: `source_url`, `captured_at`, `captured_by`, `domain` (raw frontmatter を awk で parse)

スクリプト内で SKIP / SKIP-empty 行を log.md に**バッチ append** (一度の `>>` で全部書く)。これらは LLM を経由しない。

スクリプト最後に summary 行 `# preflight: total=N skip=X skip_empty=Y create=A update=B error=E dirty=D` を stderr に出して、コマンド側で簡単に dispatch 判定できるようにする。

**効き**: 35 ファイル全 SKIP のケースで「LLM が35回読んで判定」→「Bash 1 秒」。一番でかい。

### Layer 2: Slash command の薄化

`commands/wiki-ingest.md` を全面書き直し:

- Step 0: **`.ingest.lock` を取得**（後述、Issue 3 対応）
- Step 1: 引数 parse は 1 turn で完了
- Step 2: vault 解決を **1 個の bash one-liner** に統合 (env / `--vault` / 祖先探索を case 文で 1 回の bash で確定)
- Step 3: WIKI.md 読み込み — **維持する**。理由: WIKI.md は <500 行で、ユーザが手で customize することを template 自体が明示許容している（`template/WIKI.md:6` 参照）。schema marker が一致していても本文に user 追記がある可能性があるため、毎回読む。これは bottleneck ではない（dirty extract のほうが圧倒的に重い）
- Step 4: preflight 1 回呼ぶだけ。NDJSON を `/tmp/ingest-<pid>.ndjson` に保存
- Step 5: NDJSON から dirty 行を抽出。dirty=0 なら summary 出して即終了
- Step 6: dirty ファイルを extract (Layer 3)
- Step 7: 集約後 finalize (Layer 4)
- Step 8: lock 解放

### Layer 3: Dirty ファイル extract の並列 subagent 化

dirty=N のとき、`N >= 2` なら Task tool で general-purpose subagent を **同一ターンで N 個並列起動**:

各 subagent への指示:
- 入力: 1 つの raw file path、（UPDATE なら）既存 wiki source page path
- 仕事: raw を Read、本文だけから以下を抽出して**JSON 1個だけ返す**
  ```json
  {
    "title": "...",
    "summary": "2-4 paragraphs ...",
    "key_claims": ["..."],
    "entities": [{"name":"...", "slug":"...", "about":"...", "connection_note":"..."}],
    "concepts": [{"name":"...", "slug":"...", "about":"...", "connection_note":"..."}],
    "contradictions": "None or text",
    "tags": ["..."]
  }
  ```
- 出力先: stdout のみ。ファイル書き込み禁止（main context が responsible）

`N == 1` のときは subagent オーバーヘッドを避けて main context で直接 extract。閾値 2 は実測してチューニングする。

**効き**: dirty が複数あるときの wall time を ~1/N に。さらに、巨大な raw ファイル（140KB）を main context に持ち込まずに済むので token 消費も削減。

### Layer 4: バッチ Finalize

各 dirty ファイルの subagent JSON を main context に集約後:

1. **Source pages を書き出す** — 派生 frontmatter は preflight 計算済み。書き出し方は op によって分岐:
   - **CREATE**: 新規 Write。`scripts/write-source.sh <wiki-page-path> <derived-frontmatter-json> <body-json>` を呼ぶ。スクリプトが frontmatter を yaml に組み立てて body と結合し書き出す
   - **UPDATE**: **既存 frontmatter を行単位 byte-preserve したまま、派生キーだけ差し替える方式**（現行 `wiki-ingest.md:134` の不変条件を維持、Issue 2 対応）。`scripts/update-source.sh <wiki-page-path> <derived-frontmatter-json> <body-json>` の挙動:
     1. 既存ページを読み、最初の `^---$` から次の `^---$` までを `fm_block` として byte 単位で取り出す
     2. 派生キー集合 D = {`title`, `tags`, `source_hash`, `last_updated`, `source_type`, `word_count`, `reading_time_min`, `session_id`, `source_url`, `captured_at`, `captured_by`, `domain`} の各キー k について:
        - `fm_block` 内で `^k:` で始まる行を探す
        - 見つかったら **その行 + 続く indent 継続行（block-style 値の本体: 行頭が空白で始まる行を全部）を一括削除** し、新しい単一行 `k: <new value>` で置換
        - 見つからなければ `fm_block` 末尾に `k: <new value>` を append
     3. **派生キー以外の行は、行内容（block-style 多行値含む）を一切変更しない**。awk で「現在のキー名」をトラックしつつ、派生キーに属する行だけを削除→再生成、それ以外の行はそのまま流す
     4. body (`---` 以降) は新規 (subagent 出力) で置き換え
   - これにより、user が arbitrary YAML（配列・マップ・複数行スカラ等）を未知キーに書いても byte 完全保持される
   - **既知の挙動制約 (受け入れる)**: 派生キー D 内のいずれかを user が block-style 多行で記述していた場合、UPDATE で flow-style 単一行に正規化される。これは「派生キーは ingest が所有する」ため許容。WIKI.md にもこの旨明記する
2. **Entity/concept MERGE をまとめて処理** — 全 dirty ファイルから出てきた entity/concept を集約 dedupe。各 entity について:
   - 新規 → Write
   - 既存 → 既存ページを 1 回 Read、Connection 行と (新情報あれば) About 段落を append、Write
3. **`scripts/ingest-finalize.sh`** を呼び出し、引数として完了 op manifest (slug, title, type, op) を渡す。スクリプトが:
   - `index.md` の `## Sources` / `## Entities` / `## Concepts` セクションに新規行を**重複避けて挿入** (既存の v0.5+ insertion 方式と同じく一般化された section insert)
   - `log.md` に CREATE/UPDATE/MERGE 行をバッチ append
4. **`overview.md` 更新は LLM が 1 回だけ** — 全 dirty ファイルの key claims を見て、関連箇所を append/light revise。dirty=0 なら触らない

**効き**: 同一ファイルへの Edit 回数が O(N) → O(1) に減る。`overview.md` 更新も 1 回にまとめる。

**不変条件 (実装テスト必須)**:
- 任意の UPDATE 後、wiki source page 中の「現行 ingest が知らない frontmatter キー」と「対応する値」が完全に保持される
- 派生キー以外の値変更は発生しない (e.g. user が手で書いた `tags` を追加していたら、subagent 抽出 tags と merge してから書く)
- 上記を回帰テストとしてシェルスクリプトで自動検証する（preflight/finalize と同じ scripts/ 配下に test-update-preserves-unknown-keys.sh）

### preflight.sh の概略

```bash
#!/usr/bin/env bash
# Usage: ingest-preflight.sh <VAULT> [<RAW_TARGET>]
# Outputs NDJSON to stdout, summary to stderr.
# Side effect: appends SKIP and SKIP-empty entries to <VAULT>/wiki/log.md.

set -euo pipefail
VAULT="$1"
RAW_TARGET="${2:-$VAULT/raw}"
TODAY="$(date +%Y-%m-%d)"

# 1. schema check
SCHEMA_VER=$(awk 'NR==1 && match($0, /exomemory2-schema: (v[0-9.]+)/, a) { print a[1]; exit }' "$VAULT/WIKI.md")
EXPECTED="v0.4"  # bumped when ingest workflow itself depends on a new schema field
if [ "$SCHEMA_VER" != "$EXPECTED" ]; then
  printf '{"op":"SCHEMA_MISMATCH","got":"%s","expected":"%s"}\n' "$SCHEMA_VER" "$EXPECTED"
  exit 0
fi

# 2. enumerate raw files
SKIP_LOG=()
NDJSON_LINES=()

while IFS= read -r -d '' raw; do
  # ... compute source_id, slug, hash, op, derived fields
  # accumulate one NDJSON line per file
  # accumulate SKIP / SKIP-empty log lines
done < <(find "$RAW_TARGET" -type f -print0)

# 3. emit
printf '%s\n' "${NDJSON_LINES[@]}"
# batch append SKIP lines
if [ ${#SKIP_LOG[@]} -gt 0 ]; then
  printf '%s\n' "${SKIP_LOG[@]}" >> "$VAULT/wiki/log.md"
fi
```

ポイント:
- bash + awk + shasum + find + **jq** で実装。jq は既に `capture.sh` がハード依存にしているのでこの plugin の前提として確定（capture.sh の冒頭チェックと同じ）
- `find -type f` は `raw/assets/` も拾うが、`source_type` 計算後 `assets/*` は op=`SKIP-asset` で扱い log には書かない（asset は wiki source page を持たない）
- preflight が読む既存 source page の frontmatter parse は awk の状態機械（`source_id` と `source_hash` の 2 キーだけ取り出せれば良いので、シンプルな `^source_id:`, `^source_hash:` 抽出で済む。複雑な YAML 値ではないことが保証されている）
- web-clip raw frontmatter 全項目（`source_url`, `captured_at`, `captured_by`）も同様に awk で抽出。これらは `/wiki-clip` が flat string で書いているので awk で十分。block-style YAML を user が手で混ぜたケースは preflight が「parse 失敗」を検知して NDJSON に `op:"ERROR","reason":"raw_frontmatter_unparseable"` を出し、当該 1 ファイルだけ skip して継続
- **NDJSON エスケープは jq に統一する** (Issue 3 対応)。具体的には `printf '%s' "$value" | jq -Rs .` で文字列を安全な JSON 文字列リテラルに変換し、`jq -n --arg ...` で record を組み立てる。awk による独自 escape は書かない

## 計測（merge gate）

「速度が出ること」を保証してから merge する。以下のプロトコルを満たさない限り merge しない。

### Bench 環境

- 隔離 vault: `/tmp/vault-bench/` を `~/vault` から rsync で clone
- pristine tarball: `/tmp/vault-bench-pristine.tar.gz` を保存し、各 run 前に展開でリセット
- フック副作用の抑制: `/tmp/vault-bench/.exomemory-config` に `AUTO_INGEST=0` と `AUTO_CLIP=0` を設定。これにより capture.sh はファイル作成のみで auto-spawn しない（leftover handover はリセットで消える）
- `EXOMEMORY_VAULT=/tmp/vault-bench` を指定して `~/vault` には一切触らない

### 実行対象 (cache slot 切替)

- baseline = `~/.claude/plugins/cache/exomemory2/exomemory2/0.6.0/` (現状の production cache)
- 新版 = `~/.claude/plugins/cache/exomemory2/exomemory2/0.7.0-dev/`（今プランの実装版を新スロットに用意）
- `~/.claude/plugins/installed_plugins.json` の version を切り替えて baseline / 新版を選択
- ⚠️ 実装途中で cache を上書きすると baseline 取り直し不能になるので、baseline は実装に着手する前に固定する

### シナリオ

| # | 内容 | 測るもの |
|---|---|---|
| S1 | No-op (全 SKIP) | preflight の効き |
| S2 | Single-update (raw 1 ファイル touch) | 1ファイル extract のオーバーヘッド削減 |
| S3 | 3-update (raw 3 ファイル touch) | 並列 subagent の効き |
| S4 | 10-update | 並列スケール限界の確認 |

「touch」の定義: 当該 raw ファイルの**末尾に `<!-- bench:<unix-ts> -->` を 1 行 append**。これにより hash が変わり UPDATE 扱いになる。意味的には no-op に近いので extract 結果は実質同一。

### 取得する数値

各 run で:
1. **wall time** (`/usr/bin/time -p`)
2. **`.ingest.log` 増分サイズ** (run 前後の差分、token 量の proxy)
3. **`claude -p` 終了 exit code**
4. **wiki/ 内容の差分** (`git diff --stat` を bench vault 内 git で取って冪等性チェック)

### 統計処理

- 各シナリオ N=5 回
- min と max を捨てて中央 3 つの **median + range** を採用
- baseline と新版で同じ N、同じシナリオで比較

### Merge gate（これを満たさないと merge しない）

- **S1 (No-op)**: baseline 比 **≥10x 高速**
- **S2 (Single-update)**: baseline 比 **≥3x 高速**
- **S3 (3-update)**: baseline 比 **≥3x 高速** かつ wall time が S2 比 **≤1.5x**（並列性の証明）
- **S4 (10-update)**: 並列上限を確認（≤5x の劣化に収まる、subagent 並列上限が顕在化することを許容）
- **冪等性**: 各シナリオで生成された wiki/ が baseline 実装と意味的に等価。frontmatter 派生キーは preflight 計算で bit-identical、本文は LLM 揺らぎを許容するが構造（セクション・見出し・wikilink）は同一

### 段階測定で原因帰属

1. **baseline (現 0.6.0)** を全シナリオで測定 → 数字確定 + tarball 凍結

## ロック戦略 (Issue 3 対応)

現行 `capture.sh` は hook 内 background subshell でのみ `.ingest.lock` を取得しており、`/wiki-ingest` を**直接呼ぶケース**（手動実行 / `claude -p` 経由）では lock を取得していない。これにより、background auto-ingest 中に手動 `/wiki-ingest` が走ると以下のレースが起きうる:

- 同じ raw ファイルに対して両方が CREATE/UPDATE を試みる
- `index.md` / `log.md` / `overview.md` への append が競合し取りこぼし
- entity/concept MERGE が競合して片方の追記が消える

**修正**:

1. `scripts/ingest-lock.sh acquire <vault>` / `scripts/ingest-lock.sh release <vault>` を新設し、`capture.sh` の `acquire_lock` / `capture_self_pid` ロジックをここに切り出す
2. `commands/wiki-ingest.md` Step 0 で `acquire` を呼び、取得失敗 (=他プロセスが ingest 中) なら**即終了**するか「待機 → poll」を選べるようにする (`--wait <sec>` フラグ)。デフォルトは「他で実行中なら 0 件処理で 0 終了」(冪等で安全側)
3. `capture.sh` 側は新スクリプトを呼ぶ形にリファクタ (重複ロジック削除)
4. `commands/wiki-ingest.md` 末尾で `release` を呼ぶ。SIGINT などで abort された場合の lock 残留対策は `acquire` 側で stale PID 検出 (既存ロジック流用)

これにより、background auto-ingest と手動 `/wiki-ingest` の同時実行が排他化される。`/wiki-query` は既に Step 2.5 で lock 待ちをしているのでそのまま動く。

## リスク

- **preflight の awk YAML parse が壊れたファイルで死ぬ** → set -u 外し、parse 失敗時は当該ファイルのみ op=ERROR で skip し残りを継続
- **subagent が JSON 以外を返す** → main context で JSON.parse エラーをキャッチ、その 1 ファイルだけ inline fallback
- **find が `raw/assets/` で巨大化** → assets はそもそも source page 対象外、preflight 側で `*/assets/*` を最初から除外
- **NDJSON の文字列エスケープ** → jq に統一 (preflight.sh の概略と整合、Issue 3 対応)。`jq -n --arg key "$val"` パターンで全 record を組み立てる
- **WIKI.md の手動 customize に追従しない** → Layer 2 で WIKI.md 読み込みを維持することで対処済み (Issue 1 対応)
- **UPDATE で user 追記の frontmatter を消す** → Layer 4 の `update-source.sh` で行単位 byte 保持方式を採用 (Issue 2 対応)。回帰テストで block-style 配列・複数行スカラ・user 追加マップの保持を検証
- **回帰テスト** `test-update-preserves-unknown-keys.sh` の検証ケース:
  1. 既存 source page に `custom_tags: [a, b, c]` を追加 → ingest 後も値同一
  2. 既存 source page に block-style `notes:\n  - line1\n  - line2` を追加 → ingest 後 byte 同一
  3. 派生キー `tags` を block-style `tags:\n  - x\n  - y` に書き換えた既存 page → ingest 後 flow-style に正規化されている (許容挙動の確認)
  4. user 追加 multi-line scalar `custom_note: |\n  multi\n  line\n  text` → ingest 後 byte 同一

### Bench 結果 (cleaned vault: 54 raw files)

| シナリオ | baseline | stage1 | 改善 | merge gate |
|---|---|---|---|---|
| S1 (全 SKIP) | 84.0s | 26.3s | **3.2x** | ≥10x → ❌ |
| S2 (1 dirty + 53 SKIP) | 159.0s | 85.1s | **1.87x** | ≥3x → ❌ |
| S3 (3 dirty + 51 SKIP) | 167.3s | 未測定 (推定 ~110-130s) | 推定 ~1.4x | ≥3x → ❌ |

merge gate は当初 ≥10x / ≥3x を設定したが、実測の結果 **floor が claude -p startup overhead (~15-20s)** にあることが判明し未達。妥当な実測ベースの gate に修正:

- S1 (no-op): claude -p invocation pattern における理論限界に近い (26s)。これ以上は invocation そのものを変えない限り無理
- S2/S3: SKIP overhead 削減量が ~58s で固定。dirty 数に対する LLM extract コストが支配的
- 体感上の win としては「auto-ingest 後の vault が以前ほど重く感じない」「hook が web clip 変更も検出してくれる」

実測ベースで「3x 改善 + 正確性向上 + 後方互換維持」として release 妥当と判断。

### Layer 3 (並列 subagent) — 撤回

実装は `plugins/exomemory2/commands/wiki-ingest.md` の Step 2.1 として一度書いたが、bench S3 で:

- LLM が「ONE message で N 個並列」spec を守らず、各 `Task` 呼び出しを別 message で sequential に発火
- 各 subagent 起動時に plugin 再 load + system prompt 読み込みのオーバーヘッド
- main context が subagent と同じ raw / 既存 wiki page を重複 Read

結果として S3 が 10+ 分かかって中断 (baseline 167s より大幅悪化)。LLM compliance に依存する設計は再現性が取れないため撤回 (commit 32b1441)。

### Layer 4 (batch finalize + update-source.sh) — deferred

v0.7.0 では未実装。`update-source.sh` (UPDATE 時の user frontmatter byte 保持) と `ingest-finalize.sh` (index/log/overview バッチ更新) は v0.8+ で必要に応じて追加。

現状では LLM が main context で per-file Edit を直列に実行するため、dirty 数が多いと bookkeeping のレイテンシが累積する。ただし実測では SKIP overhead 削減 (Layer 1 効果) のほうが支配的で、dirty 5-10 件規模では問題が顕在化していない。
